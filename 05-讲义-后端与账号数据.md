# 讲义 05 · 后端 · 账号 · 数据 · AI 代理（核心思路 + 具体代码）

> Day 6 用。前面都在浏览器里（前端），这一讲跨到**服务器端**。文件在 `functions/api/`，每个 `.js` 就是一个后端接口。代码总量很小（5 个文件加起来约 320 行），但概念密度高。

---

## 一、核心思路：为什么必须有后端？

前端代码是**公开的**——任何人打开网页、按 F12 都能看到 `index.html` 的全部源码。所以有三件事**绝不能放前端**：

1. **API Key**（调 AI 的密钥，等于钱）：放前端 = 把银行卡密码贴在门口。必须藏在服务器。
2. **限额 / 鉴权**：前端的判断用户能绕过，服务器的判断绕不过。
3. **数据库密码 / 别人的数据**：只能服务器碰。

所以后端的本质就是一个「**可信中间人**」：前端把请求发给它，它验明身份、查限额、拿着藏好的 Key 去问 AI 或数据库，再把结果转回来。

## 二、这套后端跑在哪：Cloudflare Pages Functions（Serverless）

你不用租服务器、不用管运维。`functions/api/generate.js` 里导出一个 `onRequestPost` 函数，Cloudflare 就会**自动**把「POST 请求 `/api/generate`」路由到它。这叫 **Serverless（无服务器）/ 函数即服务**：你只写「收到请求怎么处理」，平台负责跑起来、扩容、计费。

约定：
- 文件名 = 路由。`functions/api/auth.js` → 接口 `/api/auth`。
- 文件名以 `_` 开头（如 `_auth.js`）**不对外**，是给别的文件 import 的共享工具。
- 导出 `onRequestPost` / `onRequestGet` / `onRequestDelete` 分别处理 POST/GET/DELETE。

存储也用 Cloudflare 的：
- **D1**：一个 SQL 数据库（存 `users` 用户、`projects` 项目），代码里通过 `env.DB` 访问。
- **KV**：一个键值仓库（存「今天某用户用了几次」），通过 `env.USAGE_KV` 访问。
- **环境变量**：`env.DEEPSEEK_API_KEY`、`env.SESSION_SECRET`、`env.ACCESS_CODE` 等——**密钥都在这里，不在代码里**。

## 三、四个接口逐个看

### 1）`_auth.js` — 共享的安全工具箱（56 行，最该精读）

这里有全后端的安全核心。三块：

**a. 密码怎么存（`hashPassword`）**——绝不存明文。

```javascript
export async function hashPassword(password, saltHex){
  const key = await crypto.subtle.importKey("raw", enc.encode(password), "PBKDF2", false, ["deriveBits"]);
  const bits = await crypto.subtle.deriveBits(
    {name:"PBKDF2", hash:"SHA-256", salt:hexToBuf(saltHex), iterations:100000}, key, 256);
  return bufToHex(bits);
}
```

- **哈希（hash）**：把密码经过单向函数变成一串乱码，**不可逆**。数据库里只存这串乱码。登录时把用户输入再哈希一遍，比对两串乱码是否相同。就算数据库泄露，也拿不到原始密码。
- **salt（盐）**：每个用户一串随机值，和密码一起哈希。作用是让「相同密码」也算出不同哈希，防止黑客用「彩虹表」批量破解。
- **PBKDF2 + 10 万次迭代**：故意让哈希算得「慢」（迭代 10 万次），使暴力猜密码代价极高。

**b. 登录令牌怎么防伪造（`signToken` / `verifyAuth`）**——无状态签名令牌。

```javascript
// 令牌格式：userId.用户名.过期时间.签名
export async function signToken(env, userId, username){
  const exp = Date.now() + 30*24*3600*1000;                 // 30天后过期
  const payload = userId + "." + encodeURIComponent(username) + "." + exp;
  const sig = await hmacHex(env.SESSION_SECRET, payload);    // 用密钥对前三段签名
  return payload + "." + sig;
}
export async function verifyAuth(request, env){
  const token = ...;                                          // 从请求头 Authorization 取
  const [uid, uname, exp, sig] = token.split(".");
  const expect = await hmacHex(env.SESSION_SECRET, uid+"."+uname+"."+exp);
  if(sig !== expect) return null;                            // 签名不符 → 伪造，拒绝
  if(Date.now() > +exp) return null;                         // 过期 → 拒绝
  return { userId:+uid, username: decodeURIComponent(uname) };
}
```

核心思想（**HMAC 签名**）：令牌 = 「明文信息（谁、何时过期）+ 一段用服务器密钥算出的签名」。

- 服务器**不需要存**任何 session。验证时，用同一个密钥 `SESSION_SECRET` 重新算一遍签名，和令牌里带的签名比对：一致就说明「这是我签发的、没被篡改」。
- 黑客即使看到令牌格式，改了 userId 也算不出正确签名（因为不知道 `SESSION_SECRET`），所以伪造不了。
- 这就是「**无状态鉴权**」：服务器不记谁登录了，全靠签名自证。前端每次请求把令牌放在 `Authorization: Bearer xxx` 头里带上（对应前端 `authHeaders()`，讲义 02 第五节）。

### 2）`auth.js` — 注册 / 登录（46 行）

一个 `onRequestPost`，按 `body.action` 分「注册」和「登录」：

- **注册**：先校验用户名格式、密码长度；**校验邀请码** `body.invite === env.ACCESS_CODE`（没有邀请码不能注册）；查重；生成 salt、算哈希；`INSERT` 进 `users` 表；`signToken` 发令牌。
- **登录**：按用户名查出该用户的 salt 和存的哈希；把输入密码用**同一个 salt** 再哈希；比对；一致就发令牌。

```javascript
const u = await env.DB.prepare("SELECT id, pass_hash, salt FROM users WHERE username=?").bind(username).first();
const hash = await hashPassword(password, u.salt);
if(hash !== u.pass_hash) return json({ok:false, error:"用户名或密码错误"}, 401);
```

注意 `.prepare("...WHERE username=?").bind(username)`——**参数化查询**，用 `?` 占位再 `bind`，是防「SQL 注入」的标准写法（绝不把用户输入直接拼进 SQL）。

### 3）`projects.js` — 云端项目库（58 行）

一个文件三个接口：`onRequestGet`（列表/读取）、`onRequestPost`（新建/更新）、`onRequestDelete`（删除）。每个都先 `verifyAuth` 确认登录。**贯穿始终的一条安全线**：所有 SQL 都带 `AND user_id=?`：

```javascript
"SELECT ... FROM projects WHERE id=? AND user_id=?"     // 只能读自己的
"UPDATE ... WHERE id=? AND user_id=?"                   // 只能改自己的
"DELETE FROM projects WHERE id=? AND user_id=?"         // 只能删自己的
```

这保证「**每人只能碰自己的项目**」，就算知道别人的项目 id 也读不到。前端 `cloudSaveNow`（编辑时防抖 1.2 秒自动存）、`openProjectsPanel`（我的项目列表）就是调这里。

### 4）`generate.js` — AI 代理 + 限额（73 行）★重点

这是「可信中间人」的最典型例子。流程：

```javascript
const user = await verifyAuth(request, env);           // 1. 必须登录
if(!user) return json({error:"未登录..."}, 401);

const USER_DAILY_LIMIT = 120;                          // 2. 查今日用量（KV）
const kvKey = "usage:" + today + ":" + user.userId;
let used = parseInt((await env.USAGE_KV.get(kvKey))||"0", 10);
if(used >= USER_DAILY_LIMIT) return json({error:"今日次数已达上限"}, 429);

const upstream = await fetch("https://api.deepseek.com/chat/completions", {  // 3. 拿藏好的Key去问AI
  headers:{ "Authorization":"Bearer " + env.DEEPSEEK_API_KEY },
  body: JSON.stringify({ model:"deepseek-chat", messages: dsMessages, stream: wantStream, ... }),
});

await env.USAGE_KV.put(kvKey, String(used+1), {expirationTtl: 60*60*24*2}); // 4. 用量+1

if(wantStream) return new Response(upstream.body, {headers:{"Content-Type":"text/event-stream"}}); // 5. 流式透传
else { const data = await upstream.json(); return json({content:[{type:"text", text}]}); }         // 或整段返回
```

五步：**验登录 → 查限额 → 用服务器藏的 Key 调 DeepSeek → 计数+1 → 把结果转回前端**。

- API Key `env.DEEPSEEK_API_KEY` 只在这里出现，前端永远看不到。
- 限额按 `usage:日期:用户id` 存 KV，设 2 天自动过期（`expirationTtl`），天然实现「每天清零」。
- 流式（`wantStream`）时直接把上游的 SSE 流 `upstream.body` 透传给前端（对应讲义 04 前端的流式解析）。

## 四、数据长什么样：`outlines.js` 和数据库

- `outlines.js`（前端内置数据，不是后端）：报告大纲的兜底数据。每个领域是若干章，每章若干子标题，子标题带 `numeric` 标记（是否涉及数字）。讲义 04 说的「哪些子标题要喂真实数字」就看这个标记。
- **大纲还有一条「云端可编辑」的线**：`admin.html`（管理员后台页）→ 调 `/api/outlines`（`functions/api/outlines.js`）→ 存进 D1。前台 `index.html` 的 `fetchOutlines` **优先读云端大纲**，读不到才退回内置的 `outlines.js`。`/api/outlines` 里有个 `isAdmin()` 用环境变量 `ADMIN_USERS`（逗号分隔的用户名）判断谁能写，普通登录用户只能读。这样改章节结构不用改代码、不用重新部署。
- 数据库两张表（在 `schema.sql` 里建，部署时手动执行；**压缩包里没带这个文件**，见 `部署说明.md`）：`users`（id、用户名、密码哈希、salt）、`projects`（id、user_id、name、data 是整个项目状态的 JSON 字符串、updated_at）。**项目内容整包以 JSON 存进一个字段**，简单粗暴但够用——这就是「项目库」真正存数据的地方。

## 五、一次「登录并保存」在前后端之间怎么走

把讲义 02、05 连起来：

```
前端 showLoginModal 填账号密码
   └─ fetch POST /api/auth {action:"login", ...}
        └─ auth.js: 查库→比对哈希→signToken 发令牌
   ◀─ 返回 {token, username}
前端 setAuth 把 token 存进 localStorage，之后每次请求都带 authHeaders()

前端编辑项目 → scheduleCloudSave（防抖1.2s）→ cloudSaveNow
   └─ fetch POST /api/projects {id, name, data}  头里带 token
        └─ projects.js: verifyAuth 验令牌→ INSERT/UPDATE WHERE user_id=本人
   ◀─ {ok:true} → 顶部显示“已保存到云端”
```

## 六、Day 6 自测题

1. 为什么 API Key 必须放后端？前端放不行的根本原因是什么？
2. 密码为什么存哈希而不是明文？salt 和「10 万次迭代」各防什么？
3. 令牌（token）是怎么做到「服务器不存 session 也能判断真伪」的？改了令牌里的 userId 为什么会被识破？
4. `projects.js` 每条 SQL 都带 `AND user_id=?`，去掉会有什么后果？
5. `generate.js` 的五步是什么？限额用 KV 的 key 怎么设计的、怎么实现「每天清零」？
6.（连接题）前端 `authHeaders()`（讲义 02）和后端 `verifyAuth()`（本讲）是怎么配对工作的？

<details>
<summary>参考答案要点</summary>

1. 前端源码对用户完全公开（F12 可见），Key 放前端等于公开泄露，任何人都能盗用你的额度花钱。
2. 防数据库泄露后密码被直接读取；salt 让相同密码得到不同哈希、废掉彩虹表；高迭代次数让暴力破解代价极高。
3. 令牌自带「明文+用密钥算的 HMAC 签名」，验证时用同密钥重算签名比对，一致即未被篡改；改 userId 后签名对不上，且黑客没有 `SESSION_SECRET` 无法伪造正确签名。
4. 用户能读/改/删别人的项目（越权），只要猜到 id 即可。
5. 验登录→查限额→用藏好的Key调DeepSeek→计数+1→转回结果；KV key = `usage:日期:用户id`，设 2 天过期，换一天 key 变了、天然清零。
6. 前端把 localStorage 里的 token 放进 `Authorization: Bearer` 头（`authHeaders`），后端 `verifyAuth` 从该头取出并验签，配对完成鉴权。
</details>
