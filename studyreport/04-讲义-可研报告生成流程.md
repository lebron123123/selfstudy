# 讲义 04 · 可研报告生成流程（核心思路 + 具体代码）

> Day 5 用。这条路 = `appMode='report'`，靠 `currentStep`（0~5）走六步。核心难点在**第 5 步「AI 逐章生成」**：怎么拼提示词、怎么保证不编造财务数字、流式返回怎么解析、最后怎么导出 Word。

---

## 一、核心思路：六步流水线

`STEPS`（第 535 行）定义了六步，每步一个 `stepXxx()` 函数（由讲义 02 的 `renderSheet` 调度）：

| currentStep | 函数（行号） | 用户在干什么 | 关键产物 |
|---|---|---|---|
| 0 | `stepDomain`（968） | 选报告领域（保障房/改造…） | `domainKey` + 加载章节大纲 |
| 1 | `stepProjectInfo`（996） | 填项目名称/地点/概况、传参考资料 | `project{}` + `kbEntries[]` |
| 2 | `stepCalc`（1023） | （可选）做财务测算 | `calcResult`（喂给 AI 的真实数字） |
| 3 | `stepChapters`（1240） | 勾选要生成哪些章节 | `chapters[].checked` |
| 4 | `stepGenerate`（1252） | 逐章调 AI 起草 | 每个 `section.content` |
| 5 | `stepReview`（1520） | 人工复核、签发、导出 Word | `.docx` 文件 |

数据是**一步步累积**在全局状态里的：领域定 `chapters` 结构 → 项目信息和测算结果定「喂给 AI 的素材」→ 生成把素材变成正文 → 复核导出。

## 二、第 4 步细看：AI 逐章生成

### 1）先搭「空骨架」（`stepGenerate`, 1252 行）

进入这步时，先把勾选的章节画成一个个「占位块」（带灰色骨架条 `skel`，`section-block pending`），还没有内容。同时估算 token 消耗和费用显示给用户。此时页面是「待生成」状态。

### 2）并行起草（`runGeneration`, 1267 行）★重点

点「开始逐章生成」后：

```javascript
const tasks = [];
active.forEach(c=>c.sections.forEach((s,si)=>tasks.push({c,s,si})));  // 把所有子标题拍平成任务队列

const CONCURRENCY = 3;                                                 // 同时最多3路
const workers = Array.from({length:Math.min(CONCURRENCY,total)}, ()=>(async ()=>{
  while(tasks.length){ const t = tasks.shift(); await handleOne(t); } // 每个worker不停从队列取一个来做
})());
await Promise.all(workers);                                            // 等三路全部做完
```

这是一个经典的「**worker 池 / 生产者-消费者**」模式：

- `tasks` 是一个任务队列（每个子标题一个任务）。
- 开 3 个 `worker`，每个都在 `while(tasks.length)` 里不停 `tasks.shift()`（取出队首）来处理，直到队列空。
- `Promise.all(workers)` 等三路都结束。

好处：3 个子标题同时找 AI 写，比一个个排队快 3 倍，又不至于一次并发太多被限流。

每完成一个（`handleOne`），就把返回文本 `renderContent` 成 HTML 填进对应块，去掉骨架、盖个「已拟」章，并 `saveDraft()` 存档。失败的给「重试」按钮。

### 3）拼提示词（`generateSection`, 1447 行）★★最关键

这个函数决定「AI 写得好不好、会不会编数字」。它拼两段提示词：

- **system（角色 + 铁律）**（1455 行）：设定「资深工程咨询工程师」身份，并下几条硬规则，最重要的两条：
  1. 不得编造项目未提及的事实（地名、单位、政策文号）。
  2. 涉及金额/IRR/NPV 等：**若提供了【真实财务测算结果】就严格引用、不得改动；否则一律标"待填"，绝不编造精确数字。**
- **user（本次要写什么）**（1456 行）：把 `project{}` 里的项目信息 + 当前章节/子标题 + `kbRetrieve(...)` 检索到的相关参考资料，拼成一段「请撰写 XX 子标题」。

**「不编造数字」是怎么做到的？** 看 1448–1454 行：

```javascript
const digest = s.numeric ? buildCalcDigest() : null;   // 只有“数据类”子标题才带测算摘要
if(s.numeric && digest){
  tableHint = '...这是本项目真实计算出的结果，请严格依据这些数字撰写，不得改动、不得编造...' + digest;
} else if(s.numeric){
  tableHint = '...涉及数字但没有测算结果，一律填"待填"，绝不编造精确数字...';
}
```

- 每个子标题有个 `numeric` 标记（大纲里定义，见 `outlines.js`）。
- 若这个子标题「涉及数字」且用户做过测算 → 把 `buildCalcDigest()`（1337 行，把 `calcResult` 整理成文字+表格摘要）塞进提示词，命令 AI 只准引用这些真实数字。
- 若「涉及数字」但没测算 → 命令 AI 全部写"待填"。

这就是产品的「**AI 辅助但不瞎编**」护城河：真实数字来自 `*calc.js`（讲义 03），AI 只负责「用文字把数字讲成分析」。

### 4）流式接收（`generateSection` 1466–1490 行）

`stream:true` 时后端返回的是 **SSE 流**（服务器一段段推文字，边生成边显示，不用等全部写完）。前端逐块读、拼接、回调 `onChunk(full)` 实时刷新界面：

```javascript
const reader = resp.body.getReader();               // 拿到可读流
while(true){
  const {done, value} = await reader.read();        // 读一块
  if(done) break;
  buf += decoder.decode(value, {stream:true});      // 累积成文本
  const lines = buf.split("\n"); buf = lines.pop(); // 按行切，最后半行留着
  for(const line of lines){
    if(!line.startsWith("data: ")) continue;        // SSE 每行以 "data: " 开头
    const j = JSON.parse(line.slice(6).trim());     // 解析这一小段 JSON
    const delta = j.choices?.[0]?.delta;            // 取出增量文字
    if(delta?.content){ full += delta.content; onChunk(full); } // 拼上去并刷新
  }
}
```

理解要点：**流式 = 一小块一小块地到，前端负责把它们按 `data: ` 行拆开、拼回完整文本。** 这段和后端 `generate.js` 的「透传 SSE 流」是配套的（讲义 05）。

### 5）文字转 HTML（`renderContent`, 1315 行）

AI 返回的纯文本里，表格用 `[[TABLE]]...[[/TABLE]]` 包裹、竖线 `|` 分列。`renderContent` 用**正则**把这种标记转成真正的 `<table>`，其余按空行分段成 `<p>`。

```javascript
const tableRe = /\[\[TABLE\]\]([\s\S]*?)\[\[\/TABLE\]\]/g;   // 匹配表格标记
// 每行按 | 切成单元格，第一行当表头(th)，其余当td
```

（`escapeHtml` 是安全处理：把 `<`、`&` 转义，防止 AI 文本里的符号破坏页面。）

## 三、第 6 步：复核与导出 Word

`stepReview`（1520 行）展示归档卡片、允许「签发」（`signed=true`）。导出走 `exportWord`（1612 行）：

- `ensureDocxLib`（1549 行）：**按需加载**第三方库 `docx.umd.js`（`loadScript` 动态插 `<script>`），不到导出这步不加载，省流量。
- `buildExportPayload`（1579 行）：把所有章节内容整理成结构化数据。
- `htmlToBlocks`（1558 行）：把正文 HTML 转成 docx 库认识的「段落/表格块」。
- 最终用 docx 库生成 `.docx` 让浏览器下载。

这里不用抠 docx 库内部，抓住流程即可：**内存里的章节内容 → 转成 docx 块 → 库打包成 Word 文件**。

---

## 四、把这条路和别的模块连起来

- **和财务引擎（讲义 03）的接口**：唯一连接点是 `calcResult`。第 2 步测算把结果存进 `calcResult`，第 4 步 `buildCalcDigest()` 把它变成给 AI 的素材。**AI 从不自己算钱，钱都是 `*calc.js` 算好的。**
- **和后端（讲义 05）的接口**：`generateSection` 的 `await fetch("/api/generate", ...)`。前端只管拼提示词、发出去、收流；真正调 AI、管限额、藏 API Key 的是后端。

## 五、Day 5 自测题

1. 六步分别产出什么？哪一步的产物是「喂给 AI 的真实数字」？
2. `runGeneration` 里的「3 路并行」是怎么实现的？为什么不一次全并发？
3. AI 怎么被约束「不编造财务数字」？`numeric` 子标题在有/无测算结果两种情况下，提示词有何不同？
4. 流式返回时，前端为什么要 `buf.split("\n")` 再 `lines.pop()`？（提示：网络分块不保证按整行到达）
5. `renderContent` 把 `[[TABLE]]...[[/TABLE]]` 变成了什么？用了什么手段识别？
6.（连接题）如果用户**跳过**了第 2 步测算，第 4 步生成「投资估算」子标题时会发生什么？

<details>
<summary>参考答案要点</summary>

1. 见第一节表；第 2 步 `stepCalc` 产出的 `calcResult` 是喂给 AI 的真实数字。
2. 把所有子标题拍平成 `tasks` 队列，开 3 个 worker 各自 `while` 循环 `tasks.shift()` 处理，`Promise.all` 等全部完成；限并发是为避免过多请求被上游限流、也控节奏。
3. 通过 system 铁律 + 只对 `numeric` 且有 digest 的子标题注入真实数字并命令「只准引用」；没测算时命令「一律待填、绝不编造」。
4. 因为 TCP 分块可能把一行切成两半，`lines.pop()` 把最后可能不完整的一行留在 `buf` 里等下一块补齐。
5. 变成真正的 `<table>`；用正则 `/\[\[TABLE\]\]([\s\S]*?)\[\[\/TABLE\]\]/g` 匹配、再按 `|` 切单元格。
6. `buildCalcDigest()` 返回 null，走「无测算」分支，AI 会输出结构完整但数值全为"待填"的表格与说明，不编造精确数字。
</details>
